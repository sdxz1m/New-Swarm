# Swarm Modern CN 开发方案

## 项目目标

做一个 Swarm Simulator 风格的现代中文版重制版。原版项目只作为玩法、数据、公式和文案参考，新版使用现代前端技术重新实现。

核心目标：

- 默认中文界面，文本集中管理，后续支持中英切换。
- 现代化 UI，优化资源面板、单位购买、升级、设置、存档等布局。
- 使用新技术栈，避免 CoffeeScript、AngularJS、Bower、Grunt、Ruby Compass 等旧工具链。
- 数字单位系统独立封装，支持中文单位、英文缩写、科学计数法和自定义后缀。
- 最终可以打包成纯静态网页，方便本地运行和部署。

## 推荐技术栈

- React
- TypeScript
- Vite
- 普通 CSS 或 CSS Modules
- break_eternity.js
- localStorage 存档
- JSON/TS 数据表管理单位、升级、解锁条件和文本
- Vitest 验证公式、存档和核心引擎

## 第一阶段：现代中文 MVP

目标是先做一个可玩的现代中文原型，不追求完整复刻。第一阶段的重点是“小内容量、正确架构”，而不是堆满旧版内容。

功能范围：

- 资源系统：肉、幼虫、领地等核心资源。
- 单位系统：无人虫、女王、巢穴等基础单位购买、成本递增、自动生产。
- 升级系统：少量基础升级，用于验证解锁、单位倍率和资源倍率逻辑。
- 数字格式：中文单位与科学计数法切换。
- 存档系统：自动保存、手动保存、重置、导入导出。
- 现代布局：左侧导航，中间主操作区，右侧升级/统计区。
- 响应式：桌面优先，兼顾移动端基本可用。

第一阶段完成标准：

- 打开网页即可游玩核心循环。
- 所有主要 UI 文本为中文。
- 买单位、涨资源、解锁升级、保存读档都能正常工作。
- 数值单位可以在设置中切换。

## 第二阶段：迁移原版核心内容

## 第三阶段：完整体验与打磨

任务：

## UI 方向

整体感觉：现代、清晰、有信息密度，但不要像传统后台系统那样冷。

建议布局：

- 左侧：资源摘要、导航入口。
- 中间：当前页面主内容，例如单位购买列表。
- 右侧：升级、解锁提示、统计摘要。
- 顶部：当前生产速率、存档状态、设置入口。

组件建议：

- 资源卡片：显示数量、每秒产量、容量或趋势。
- 单位行/单位卡：显示名称、说明、拥有数量、产量、价格、购买按钮。
- 升级卡：显示效果、成本、是否可购买。
- 设置弹窗：语言、数字单位、存档操作。

## 数字单位方案

数字格式做成独立模块，例如：

- `zh`: `1.23万`、`4.56亿`、`7.89兆`
- `short`: `1.23K`、`4.56M`、`7.89B`
- `scientific`: `1.23e6`
- `custom`: 从配置表读取后缀

以后修改单位时，只改格式化配置，不散落到 UI 组件里。

## 引擎边界

第一阶段就保留下面这些结构，以便后续迁移旧版数据：

- `cost`：资源成本和递增公式。
- `prod`：单位生产资源或后续扩展为单位生产单位。
- `requires`：资源、单位、升级解锁条件。
- `effect`：升级效果，例如单位产量倍率、资源产量倍率。
- `save`：新版存档版本号，旧版存档迁移暂缓。

## 数据组织方案

建议把玩法数据和 UI 代码分开。

示例：

```text
src/
  data/
    resources.ts
    units.ts
    upgrades.ts
  game/
    engine.ts
    formulas.ts
    save.ts
    numberFormat.ts
  i18n/
    zh-CN.ts
    en-US.ts
  components/
  pages/
  styles/
```

## 开发顺序

1. 初始化 Vite + React + TypeScript 项目。
2. 建立基础布局和视觉风格。
3. 实现游戏状态、tick 循环和资源增长。
4. 实现单位购买和成本公式。
5. 实现数字格式模块。
6. 实现本地存档。
7. 加入第一批升级。
8. 对照原版迁移更多数据。

## 暂不做的事

第一阶段先不做：

- 完整复刻全部原版单位和升级。
- 旧版存档兼容。
- 联机、账号、统计服务。
- 复杂动画和音效。
- 多语言完整翻译。

这些等核心结构稳定后再补。
